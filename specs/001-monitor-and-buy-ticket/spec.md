# SPEC-001: 车票监控与自动下单系统

## 1. 用户故事

**作为**一名每日通勤的上班族，
**我希望**通过本地 curl 命令启动车票监控，系统自动检测有票后创建订单，
**以便**我不需要反复手动打开微信小程序刷票，只需在收到订单后完成微信支付即可。

### 典型使用场景

```bash
# 场景一：监控本周剩余天数的车票（今天至周日）
curl -X POST http://localhost:8080/monitor/this-week

# 场景二：监控下周车票（周一至周日）
curl -X POST http://localhost:8080/monitor/next-week

# 场景三：监控指定日期的车票
curl -X POST http://localhost:8080/monitor/dates \
  -H "Content-Type: application/json" \
  -d '{"dates": ["2026-03-25", "2026-03-26"]}'
```

## 2. 功能性需求

### 2.1 监控范围

| 监控类型 | 日期范围 | 说明 |
|---------|---------|------|
| 本周 | 今天 ~ 本周日 | 已过去的日期自动排除 |
| 下周 | 下周一 ~ 下周日 | 完整7天 |
| 指定日期 | 用户指定 | 已过去的日期自动排除 |

- **实际监控日期** = 用户请求的日期范围 **交集** 「查询车次库存」接口返回的可用日期
- 已过去的日期不纳入监控

### 2.2 监控与轮询

- 默认每 **30 秒** 轮询一次「查询车次库存」接口
- 轮询间隔可配置
- 每轮监控对所有监控日期统一查询（一次请求返回所有日期的库存）

### 2.3 创建订单流程

对每个有票且未购票的日期，按以下步骤执行：

#### Step 1: 查询车次库存

- **接口**：`POST /api/v1/route/schedule/find`
- **请求**：`{"route_id": 275}`
- **判断有票**：对应日期的 `number` 字段 > 0
- 筛选出在监控日期范围内且有票的车次

#### Step 2: 检查已购票

- **接口**：`POST /api/v1/order/get`
- **请求**：`{"status": "", "page": 1, "limit": 10}`
- **匹配规则**：按 `route_name` + `description.date`（仅日期部分，如 `2026-03-24`）匹配
- **跳过条件**：`trade_state` 为「未支付」或「支付成功」的订单视为已购票，跳过该日期

#### Step 3: 查询可用优惠券

- **接口**：`POST /api/v2/order/coupon`
- **请求**：`{"route_id": 275, "schedule_ids": [61429], "boarding_point_id": 24}`
- **筛选条件**：`is_use` 为 `true` 且 `status` 为「待使用」的优惠券

#### Step 4: 验证订单价格

- **接口**：`POST /api/v2/order/price/verification`
- **逻辑**：依次尝试每张可用优惠券进行验证
  - 验证成功（`code: 200`）→ 使用该优惠券，进入 Step 5
  - 验证失败 → 尝试下一张优惠券
  - **所有优惠券均失败 → 不使用优惠券，直接进入 Step 5**

#### Step 5: 创建订单

- **接口**：`POST /api/v2/order/create`
- **请求参数**：`route_id`、`boarding_point_id`、`alighting_point_id`、`schedule_ids`、`coupon_ids`（可为空）
- **不自动支付**，用户需在微信小程序中手动完成支付

#### Step 6: 记录购票日志

- **接口**：`POST /api/v1/order/get`
- 创建订单成功后，通过返回的 `wx_order_id` 在订单列表中匹配，提取日志字段
- 写入 `./logs/ticket.log`

### 2.4 购票日志

- **日志路径**：`./logs/ticket.log`
- **写入时机**：每次创建订单成功后
- **日志字段**：
  - `date`：车票日期，来自 `description.date`
  - `route_name`：路线名称，来自 `route_name`
  - `start_stop`：上车站，来自 `description.start_stop`
  - `end_stop`：下车站，来自 `description.end_stop`
- 每张车票之间使用分割线分隔
- **每周五 10:00 清空日志文件**

### 2.5 可配置项

| 配置项 | 默认值 | 说明 |
|--------|--------|------|
| `route_id` | `275` | 监控的路线 ID（17号线-明珠线-上班） |
| `boarding_point_id` | `24` | 上车站 ID（长沙圩①） |
| `alighting_point_id` | `400` | 下车站 ID（科创中心西门） |
| `auth_token` | 手动配置 | 微信小程序 X-Auth-Token |
| `monitor_interval` | `30` | 轮询间隔（秒） |

## 3. 非功能性需求

### 3.1 架构解耦

- **API 层**与**业务逻辑层**分离：本地 curl API 负责接收请求和参数校验，业务逻辑层负责监控与下单流程
- **第三方 API 调用**封装为独立的客户端组件，便于测试时替换
- 配置项集中管理，避免硬编码

### 3.2 错误处理

- 第三方 API 调用失败（网络超时、非 200 响应）时，记录错误日志，不中断监控循环
- Token 过期（接口返回认证失败）时，记录明确的错误提示，继续下一轮监控
- 创建订单流程中任一步骤失败，跳过当前日期，继续处理下一个日期

### 3.3 日志与可观测性

- 每轮监控的关键动作（查询库存、检查已购票、尝试优惠券、创建订单）需输出控制台日志
- 错误信息包含足够的上下文（接口名称、请求参数、响应内容）

## 4. 验收标准

### AC-1: 监控本周车票

- **Given** 今天是周三（2026-03-25）
- **When** 用户调用 `POST /monitor/this-week`
- **Then** 系统开始监控周三至周日的车票，已过去的周一、周二不在监控范围内

### AC-2: 监控下周车票

- **Given** 当前是本周
- **When** 用户调用 `POST /monitor/next-week`
- **Then** 系统开始监控下周一至下周日的车票

### AC-3: 监控指定日期

- **Given** 用户指定日期 `["2026-03-25", "2026-03-26"]`
- **When** 用户调用 `POST /monitor/dates`
- **Then** 系统仅监控这两个日期的车票

### AC-4: 有票时自动创建订单

- **Given** 车次库存接口返回 2026-03-25 的 `number > 0`
- **And** 订单列表中该日期、该路线无「未支付」或「支付成功」的订单
- **When** 系统检测到有票
- **Then** 系统依次执行：查询优惠券 → 验证价格 → 创建订单

### AC-5: 已购票日期自动跳过

- **Given** 订单列表中 2026-03-25、route_name 为「17号线-明珠线-上班」已有 `trade_state` 为「支付成功」的订单
- **When** 系统检测到该日期有票
- **Then** 跳过该日期，不重复下单

### AC-6: 优惠券逐张尝试

- **Given** 有 3 张可用优惠券（A、B、C）
- **When** 优惠券 A 验证失败，优惠券 B 验证成功
- **Then** 使用优惠券 B 创建订单，不尝试优惠券 C

### AC-7: 所有优惠券失败时不使用优惠券

- **Given** 所有可用优惠券验证均失败
- **When** 系统尝试完所有优惠券
- **Then** 不使用优惠券直接创建订单

### AC-8: 购票日志写入

- **Given** 订单创建成功，返回 `wx_order_id: 572468`
- **When** 系统从订单列表接口获取该订单详情
- **Then** 在 `./logs/ticket.log` 中追加一条记录，包含 date、route_name、start_stop、end_stop，并以分割线分隔

### AC-9: 日志每周清空

- **Given** 当前时间为周五 10:00
- **When** 定时任务触发
- **Then** `./logs/ticket.log` 文件内容被清空

### AC-10: 监控间隔

- **Given** 监控间隔配置为 30 秒
- **When** 一轮监控完成后
- **Then** 等待 30 秒后开始下一轮监控

### AC-11: 第三方 API 调用失败不中断监控

- **Given** 查询车次库存接口返回网络超时
- **When** 本轮监控失败
- **Then** 记录错误日志，等待下一轮继续监控

### AC-12: 日期交集逻辑

- **Given** 用户请求监控本周（周三至周日共 5 天）
- **And** 车次库存接口只返回周三、周四、周五的数据
- **When** 系统计算实际监控日期
- **Then** 实际监控日期为周三、周四、周五（取交集）

## 5. 输出格式示例

### 5.1 创建订单成功 — API 响应

```json
{
  "code": 200,
  "msg": "购票成功",
  "data": {
    "wx_order_id": 572468,
    "is_zero_order": false
  }
}
```

```
----------------------------------------
日期: 2026-03-24 07:40:00

上车站: 长沙圩①

----------------------------------------
日期: 2026-03-25 07:40:00
路线: 17号线-明珠线-上班
上车站: 长沙圩①
下车站: 科创中心西门

```

### 5.3 本地 API 接口清单

| 方法 | 路径 | 请求体 | 说明 |
|------|------|--------|------|
| POST | `/monitor/this-week` | 无 | 监控本周（今天至周日） |
| POST | `/monitor/next-week` | 无 | 监控下周（周一至周日） |
| POST | `/monitor/dates` | `{"dates": ["2026-03-25"]}` | 监控指定日期 |

### 5.4 涉及的第三方 API

| 步骤 | 接口 | 方法 | 路径 |
|------|------|------|------|
| 查询车次库存 | schedule/find | POST | `/api/v1/route/schedule/find` |
| 检查已购票 | order/get | POST | `/api/v1/order/get` |
| 查询可用优惠券 | order/coupon | POST | `/api/v2/order/coupon` |
| 验证订单价格 | price/verification | POST | `/api/v2/order/price/verification` |
| 创建订单 | order/create | POST | `/api/v2/order/create` |
